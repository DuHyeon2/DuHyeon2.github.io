PROJECT 01

NetCDF 데이터 처리 파이프라인 최적화

디지털 트윈 플랫폼 · 대규모 다차원 배열 과학 데이터 가공 서비스

기간2025.08 ~ 2026.01
기여도100%
FastAPITemporalRedisS3KubernetesHPAJMeter

문제 상황

  • 대용량 응답으로 클라이언트 다운 — NetCDF 시계열 데이터를 한 번에 전송하니 클라이언트가 응답을 처리하지 못하고 다운되는 현상 발생, 커넥션도 장시간 점유
  • 중단 시 전체 재시작 — Worker Pod가 중간에 재시작되거나 배포되면, 이미 처리한 수천 개의 타임스텝도 처음부터 다시 처리해야 함
  • 단일 Pod 병목 — FastAPI 하나가 수신·처리·응답을 모두 담당, 수평 확장 구조 부재로 요청 증가 시 응답 지연 폭증
  • 대용량 업로드 시간 지연 — 클라이언트 → 서버 → NFS로 경유되며 서버가 전체 파일을 수신한 뒤 NFS에 저장해야 해서 업로드 완료까지 시간이 길어지고 서버 메모리·네트워크 비용도 이중으로 발생

해결 과정

  • 응답 모델 재설계 — 한 번에 전부 전송하는 구조에서, 타임스텝별 결과물을 Redis에 저장하고 클라이언트가 시계열 단위로 Key를 통해 순차 조회하는 구조로 전환
  • Temporal 비동기 처리 도입 — FastAPI는 Workflow 시작과 Key 반환만 담당, 실제 처리는 Worker가 수행 — 요청과 실행을 분리
  • Heartbeat 기반 체크포인트 복구 — 처리 중인 타임스텝 인덱스를 activity.heartbeat로 주기 보고, 재시작 시 heartbeat_details에서 마지막 인덱스를 꺼내 이어서 처리
  • S3 Presigned URL 업로드 구조 — 클라이언트가 S3에 직접 업로드 후 서비스가 S3에서 copy해 NFS에 저장 — 업로드 시간 단축 + 서버 부하 절감
  • HPA 수평 확장 — Worker Pod 수를 HPA로 오토스케일링, 부하에 따라 처리량 확보

역할

  • 아키텍처 전환 설계 — SSE 직접 전송 방식의 한계(커넥션 점유, 중단 시 재처리)를 진단하고, Temporal + Redis 기반 비동기 구조로의 전환 결정 및 설계
  • 프론트와 응답 스펙 합의 — Redis Key 반환 방식을 먼저 제안, 클라이언트 소비 방식·조회 패턴을 프론트 개발자와 협의 후 확정하고 구현 착수
  • 체크포인트 복구 구현 — heartbeat 콜백 함수를 이미지 생성 루프에 주입, Activity 재시작 시 heartbeat_details에서 start_idx를 꺼내 이어서 처리하도록 구현
  • S3 Presigned URL 발급 API 구현 — KakaoCloud Object Storage 연동, 서버에서 임시 URL을 발급해 클라이언트가 S3에 직접 업로드 → 서비스가 S3에서 copy해 NFS에 저장하는 구조 설계 및 엔드포인트 구현
  • HPA 설정 및 튜닝 — Worker Pod의 HPA minReplicas/maxReplicas 설정, cold-start 지연을 고려한 최소 Replica 수 결정
  • JMeter 부하 테스트 — 동시 50건 부하 시나리오 구성, 처리량·에러율 측정 및 HPA 튜닝 결과 검증

기술 선택 근거

SSE 직접 스트리밍 대신 Redis Key 반환을 택한 이유
SSE는 단일 커넥션으로 실시간 전송에 적합하지만, 연결이 끊어지면 클라이언트가 어디까지 받았는지 알 수 없고 재연결 시 처음부터 다시 받아야 함. Redis Key 방식은 원하는 시점에 원하는 범위만 조회할 수 있어 대용량 시계열 데이터에 더 적합하다고 판단.
단순 재시도 대신 Heartbeat를 택한 이유
타임스텝 처리 중 Worker가 죽으면 단순 재시도는 처음부터 다시. Heartbeat로 진행 상태를 저장해두면 마지막 위치부터 이어받기 가능.
서버 직접 업로드 대신 Presigned URL을 택한 이유
기존 클라이언트 → 서버 → NFS 구조에서는 서버가 대용량 파일 전체를 수신한 뒤 NFS에 저장해야 해서 업로드 시간이 길어지고 서버 메모리·네트워크 비용도 이중으로 발생. Presigned URL 방식은 서버가 인증 토큰만 발급하고 클라이언트가 S3에 직접 업로드, 서비스는 이후 S3에서 copy해 NFS에 저장만 수행하므로 업로드 소요 시간이 줄고 서버 부담 없이 확장 가능.
성과 · 2.5/min → 7.1/min
동시 50건 부하 기준 처리량 약 3배 향상. 단일 Pod → Temporal + HPA 수평 확장 구조로 전환, 에러율 0% 유지.

배운 점

장시간 반복 처리는 "재시도"가 아니라 "이어받기(체크포인트)"가 답이다. 진행 상태를 외부에 저장해두는 설계가 장애 복원력의 핵심이다.

아쉬운 점

  • heartbeat_timeout 시간 선택이 실측 기반 튜닝이 아닌 보수적 설정 수준 — 실제 heartbeat 주기 분포에 맞춘 정교한 값 조정 필요
  • Temporal 내부의 Replay·상태 저장 메커니즘은 더 깊이 이해하는 것이 다음 학습 과제

  • 질문 유형별 처리 분기 — 도메인별로 다른 프롬프트·후처리가 필요했으나 단일 체인으로는 분기 어려움
  • 노드 중간 장애 시 무한 대기 — LangGraph 노드가 AI 호출 중 멈추면 전체 워크플로우가 응답을 못 받고 무한 대기 — 자동 재시도·타임아웃 부재

해결 과정

  • 비동기 응답 패턴 적용 — 요청 즉시 Channel Key 반환, 클라이언트는 Pub/Sub 구독으로 결과 수신
  • LangGraph 도메인 라우팅 — intent 분류 노드가 사용자 질문 유형을 판별 → add_conditional_edges로 도메인별 처리 노드로 조건부 분기
  • Temporal 워크플로우 단위 재시도 — LangGraph 그래프 전체를 하나의 Temporal Activity로 감싸 자동 재시도·타임아웃·지수 백오프 적용 — 개별 LLM 호출 장애가 전체 완주를 막지 못하도록 설계, 무한 대기 문제 해소
  • Pub/Sub fan-out 구조 — 동일 결과를 여러 클라이언트가 동시 구독할 수 있도록 Redis Pub/Sub 채택

역할

  • LangGraph 노드 설계·구현 — 질문 유형 분류 노드와 도메인별 처리 노드를 직접 설계, TypedDict 기반 State 정의로 노드 간 데이터 전달
  • 조건부 라우팅 구현 — add_conditional_edges를 활용해 질문 유형별 워크플로우 분기 로직 구현
  • 비동기 응답 구조 설계 — Channel Key 반환 + Pub/Sub 구독 패턴 설계 및 API 엔드포인트 규약 정의
  • Temporal 도입 제안 — NetCDF 프로젝트에서 실운영으로 먼저 검증한 Temporal을 LLM 오케스트레이션에 도입 제안 — 외부 API 실패 복원력·타임아웃 관리 이점을 논거로 팀 합의 도출

기술 선택 근거

SSE 직접 스트리밍 대신 Channel Key 반환을 택한 이유
OpenAI API 응답이 1분 이상 소요되는 상황에서 SSE로 직접 전송하면 커넥션이 장시간 점유됨. Channel Key 반환 방식은 요청과 결과 수신을 분리해 여러 클라이언트가 동일 결과를 구독할 수 있고, 커넥션 타임아웃 문제도 해소.
단일 체인 대신 LangGraph를 택한 이유
질문 유형에 따라 다른 프롬프트·후처리가 필요한 상황에서 단일 체인은 분기 로직이 코드에 산재됨. LangGraph의 conditional_edges로 라우팅을 그래프 구조로 표현하면 분기가 많아져도 노드 단위로 관리 가능.
Temporal을 도입한 이유
LangGraph 노드 중간에 LLM 호출이 멈추면 전체 워크플로우가 무한 대기 상태에 빠짐. LangGraph 그래프 전체를 하나의 Temporal Activity로 감싸 자동 재시도·타임아웃·지수 백오프를 적용, 일시적 LLM API 장애를 흡수하는 구조로 전환.
성과 · 에러율 14% → 0%
동시 50건 부하 기준 완주율 100% 확보. LangGraph 그래프 전체를 Temporal Activity로 감싸 자동 재시도·타임아웃·지수 백오프 적용.

배운 점

  • 복잡한 LLM 워크플로우는 선형 체인보다 그래프로 모델링할 때 분기·재사용이 명확해진다 — 도메인 로직이 다양할수록 LangGraph의 노드 분리가 유지보수성을 높인다
  • 외부 API를 부르는 워크플로우는 "성공 시간"이 아니라 "실패 시 복원 전략"이 안정성을 결정한다 — 자동 재시도·타임아웃·백오프를 워크플로우 단위로 걸어두면 일시적 장애가 전체 실패로 이어지지 않는다

아쉬운 점

  • Pub/Sub 구독 전 발행 시 메시지 유실 가능성에 대한 보완 설계 미흡
  • 50건 샘플 기준 측정으로 통계적 신뢰도는 제한적 — 더 큰 샘플과 다양한 장애 시나리오에서의 지속 측정 필요
  • 실측 LLM 응답 지연 분포에 맞춘 재시도·타임아웃 값 튜닝이 미흡 — 현재 값은 경험적 설정 수준, 추후 지속 관측 필요
PROJECT 03

Kafka Streams 기반 CSV 적재 파이프라인

디지털 트윈 플랫폼 · CSV 동기 적재 → 이벤트 기반 비동기 파이프라인 전환

기간2025.07 ~ 2025.09
기여도60%
Spring BootKafkaKafka StreamsProtobufWebSocket

문제 상황

  • 동기 처리로 API 점유 — CSV 적재 요청이 FastAPI 안에서 동기로 DB INSERT까지 수행되어, 처리 동안 워커가 묶여 다른 요청 응답 지연
  • 행 단위 INSERT로 DB 부하 — 파싱된 행을 한 건씩 INSERT하는 구조로 트랜잭션 비용·네트워크 왕복이 누적되어 DB CPU에 부담
  • 중단 시 재요청 부담 · 확장 한계 — 처리 중간에 끊기면 어디까지 적재됐는지 추적이 어렵고, 적재 로직이 API 서버에 묶여 별도 스케일이 불가능

해결 과정

  • 역할 분리 — 3단계 파이프라인 — FastAPI는 파싱·검증·발행만, Streams가 변환·SQL 생성, Consumer가 실제 적재·진행률 푸시를 담당하도록 분리
  • WebSocket 진행률 푸시 — Consumer가 누적 처리 건수를 추적해 percent 계산 후 진행률 실시간 전송

역할

  • 전환 결정과 아키텍처 설계 — 기존 FastAPI 내부 동기 적재 흐름을 3단계 이벤트 파이프라인으로 분리하는 의사결정과 전체 토폴로지 설계
  • Producer 추가 — 기존 CSV 파싱 로직은 그대로 두고, 파싱 결과를 행 단위 Protobuf로 Kafka에 발행하는 Producer 레이어 추가
  • Streams · Consumer 신규 구현 — KStream 파이프라인, 상태별 분기, PostGIS WKT SQL 생성 및 Consumer Bulk Insert

기술 선택 근거

Kafka Streams를 택한 이유
단순 컨슈머로 적재만 한다면 일반 Consumer로 충분. 그러나 좌표 검증·SQL 생성·상태 분기·배치 그룹핑처럼 흐름 자체에 변환 단계가 여러 개 있어 KStream의 선언적 파이프라인 모델이 코드 구조 측면에서 더 명확하다고 판단.
Protobuf를 두 종류로 분리한 이유
입력은 "원시 CSV 행", 출력은 "DB에 던질 SQL 배치"로 의미가 다름. 하나의 proto로 묶으면 nullable 필드가 늘어나고 의미가 흐려지므로 분리. 각 토픽이 무엇을 운반하는지 타입 시그니처만으로 표현.
Bulk Insert 이유
단건 INSERT를 보내면 트랜잭션 및 네트워크 왕복이 행 수만큼 누적되지만, BatchUpdate로 한 번에 묶어 보내면 왕복 비용이 배치당 1회로 줄어 DB 부하와 적재 시간이 동시 감소.

배운 점

  • 의도에 맞게 Kafka Streams와 Kafka Consumer를 사용함으로써 Kafka에 대한 이해도가 올라갔다
  • Protobuf로 단계별 의도를 코드 구조로 표현하는 게 유지보수에 유리
  • 배치는 size만으로도 time만으로도 부족하다 — 둘을 함께 다뤄야 실운영에서 안정적이다

아쉬운 점

  • bulkInsert 실패 시 어떤 행이 실패했는지 추적·재처리하는 DLQ 전략 미흡 — 현재는 트랜잭션 전체 롤백
  • Kafka Streams 처리량을 별도 노출하지 않아 운영 중 병목 현상 발생 시 외부 확인이 미흡

문제 상황

  • 1분마다 전체 조회 — N개 센서 테이블을 1분 간격으로 전체 SELECT, 변경 여부와 무관하게 하루 43,200건 이상 반복
  • 데이터 반영 지연 — 센서 데이터 추가 후 최대 1분 후에야 화면에 반영, 실시간성 부족
  • 테이블 확장 부담 — 신규 센서 테이블 추가 시 스케줄러에 분기 로직이 누적, 단일 파일 비대화

해결 과정

  • WAL 기반 변경 감지 — PostgreSQL WAL 로그를 Debezium Embedded가 Spring Boot 앱 내에서 읽도록 구성
  • 테이블별 처리 클래스 분리 구조 — 어노테이션으로 테이블-처리 클래스를 매핑, 신규 테이블 추가 시 기존 코드 수정 없이 클래스 1개만 작성하면 연결되는 구조
  • 처리 전략 이원화 — 알람은 WebSocket 즉시 푸시, 센서 데이터는 큐에 쌓아 배치 처리로 분리
  • 동시성 안전한 이벤트 버퍼 — 여러 테이블 이벤트가 동시에 몰려도 데이터 유실 없이 처리되도록 동시성 안전한 큐 자료구조로 이벤트 버퍼 구현

역할

  • 전체 아키텍처 설계 — 폴링 → 이벤트 기반 전환 의사결정 및 Debezium Embedded 기반 구조 선택
  • 확장 가능한 처리 구조 설계 — 어노테이션 기반 매핑 구조 설계, 신규 테이블 추가 시 클래스 1개 작성으로 확장 가능하도록 구현
  • JPA 기반 도메인 모델링 — 재난안전 도메인 엔티티 간 관계를 @Entity로 설계, JPA 연관관계 매핑 직접 구성
  • WebSocket 메시지 프로토콜 정의 — 알람·센서 데이터 전송을 위한 JSON 메시지 스키마 직접 설계, 프론트엔드와 인터페이스 규약 합의

기술 선택 근거

Kafka Connect를 쓰지 않은 이유
Kafka 클러스터 운영 부담과 추가 인프라 비용을 감수할 정도의 요구사항이 아니었음. Spring Boot 앱 하나에 임베드하는 것이 운영상 단순.
DB 트리거를 쓰지 않은 이유
테이블이 늘어날수록 트리거 관리 비용 증가. WAL 기반이면 DB 레벨에서 모든 변경을 자동 캡처.
분기 로직 대신 클래스 분리를 택한 이유
테이블이 5개만 넘어가도 분기 로직이 한 파일에 쌓여 유지보수가 어려워짐. 테이블당 클래스 하나로 분리하면 신규 테이블 추가 시 기존 코드 수정이 불필요해 확장 비용이 테이블 수에 비례해 늘지 않음.
성과 · 일 폴링 쿼리 43,200 → 0
DB CPU 약 35% 감소, 변경 감지 지연 60s → 50ms.

배운 점

  • "무엇을 변경할지"가 아니라 "변경이 일어났는지"를 DB가 알려주게 하면, 애플리케이션은 반응만 하면 된다
  • 처음에 구조 잡는 시간은 나중에 몇 배로 돌려받는다 — 테이블별로 처리 로직을 분리해둔 덕분에 이후 다수의 테이블이 추가되어도 기존 코드를 수정할 필요가 없었다

아쉬운 점

  • 앱 재시작 시 오프셋 기반 복구 전략이 더 정교할 필요
  • Debezium Engine 자체 장애 시 자동 복구 로직 미흡

  • 요구사항 직접 소통 — 군청 담당자와 직접 커뮤니케이션하며 현장 요구사항 반영
  • 공공 인증 프로토콜 학습 — GPKI 인증·부동산시스템 연동 스펙을 직접 조사·적용

배운 점

  • 공공 시스템 연계는 기술 스펙보다 인증 프로토콜·정책 제약이 더 큰 변수임을 체감 — GPKI 연동에서 기술 난이도보다 절차·증빙이 더 복잡

SIDE PROJECT 06

웹툰 검색 파이프라인

수동 크롤링 + PostgreSQL → Airbyte + Elasticsearch 자동화

기간2025
기여도100%
AirbyteElasticsearchPostgreSQL

문제 상황

  • 수동 적재 부담 — 1분 주기 크론 크롤링 + SQL 비교 + DB 적재를 Shell Script로 직접 관리
  • 검색 응답 저하 — PostgreSQL LIKE 검색은 데이터 증가에 따라 응답 시간 급락

해결 과정

  • EL 자동화 도입 — Airbyte 커넥터로 PostgreSQL → Elasticsearch 스케줄 Sync 구성
  • 검색 전용 엔진 분리 — 텍스트 검색은 Elasticsearch의 역인덱스 기반으로 처리

기술 선택 근거

Airflow 대신 Airbyte를 쓴 이유
DAG 작성 없이 커넥터 기반 EL 자동화가 가능. 단순 적재 목적에 Airflow는 과함.
PostgreSQL FTS 대신 Elasticsearch를 쓴 이유
텍스트 검색·필터 위주 패턴. 역인덱스 구조가 LIKE 대비 검색 성능 우위.

배운 점

"저장"과 "조회"는 같은 DB로 해결할 필요가 없다 — 목적에 맞는 엔진을 분리하는 것이 더 단순할 때가 많다.

아쉬운 점

  • Elasticsearch의 UPDATE/DELETE 제약으로 데이터 수정 시 재적재 필요

문제 상황
