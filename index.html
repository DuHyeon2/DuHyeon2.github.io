NetCDF 파일당 수천 개 타임스텝 이미지 생성, 단일 Pod에서 평균 282초 소요로 사용자 대기 불가

재시도 로직 부재

데이터 가공 도중 에러 발생 시 전체 파이프라인 실패, 수동 재실행 필요

Worker 단위 자동 재시도 및 타임아웃 선언적 관리

Redis 적재 + Key 반환

🔧 기술 선택 근거
왜 Temporal?

워크플로우 상태 추적, 자동 재시도, 타임아웃 관리가 선언적으로 가능.

왜 Redis?

Redis에 적재하여 클라이언트 읽기 지연 최소화. TTL로 자동 만료 관리.

왜 HPA?

Temporal Worker의 max_concurrent_activities=2로 제한하되, Pod 수를 HPA로 수평 확장하여 처리량 확보.

Project 2 데이터 동기화 · 이벤트 기반

CDC 데이터 파이프라인 구축

Debezium Embedded Engine 기반 경량 CDC 구현
기간: 2023.11 ~ 2024.08 | 기여도: 100% Java Spring Boot Debezium PostgreSQL WebSocket
🎯 도전 과제 & 배경
1분마다 전체 조회

각 센서 테이블을 1분 간격으로 전체 조회하는 비효율 구조, 데이터 변경 여부와 무관하게 반복 실행

불필요한 쿼리 반복

N개 테이블 × 24시간 × 60분 = 하루 43,200건 이상 데이터가 추가되지 않아도 매분 쿼리 실행

데이터 반영 지연

센서 데이터가 추가돼도 최대 1분 후에야 화면에 반영, 실시간 대응이 필요한 상황에서 치명적

📐 AS-IS Architecture
1분마다 전체 테이블 조회

변경 여부와 무관하게 N개 테이블을 매분 전량 SELECT, DB에 불필요한 부하 지속 발생

최대 1분 반영 지연

센서 데이터가 추가돼도 다음 폴링 주기까지 대기, 실시간 대응 불가

📐 TO-BE Architecture
데이터 추가 시 자동 감지

DB에 센서 데이터가 추가되는 순간 자동으로 감지하여 처리 (PostgreSQL WAL 기반)

테이블별 핸들러 자동 매핑

각 계측기 테이블마다 처리 로직을 자동으로 매핑, 새 테이블 추가 시 핸들러 클래스만 추가하면 완료

배치/즉시 처리 분리

알람 데이터는 즉시 처리, 센서 데이터는 모아서 배치 처리

Kafka 없는 경량 구조

별도 인프라 추가 없이 Spring Boot 앱 하나로 변경 감지 운영

🔧 기술 선택 근거
왜 Embedded Debezium?

Kafka 클러스터 없이 앱 하나로 변경 감지 가능, 인프라 최소화가 핵심이었기 때문에 Embedded 방식 선택

왜 WAL 기반?

테이블마다 트리거를 수동 등록할 필요 없이, DB 레벨에서 모든 테이블의 데이터 추가를 자동 캡처

LLM 응답 지연 · 중단

OpenAI API는 응답까지 수십 초 소요, 중간에 타임아웃으로 끊기면 전체 결과 유실

클라이언트 커넥션 장시간 점유

응답 완료까지 클라이언트가 45초+ 대기, 중단 시 처음부터 재시도 필요

체인 실패 시 전체 재실행

여러 LLM 단계로 구성된 체인에서 중간 실패 시 앞 단계 결과까지 버리고 처음부터 재실행

📐 AS-IS Architecture
동기 API 호출 블로킹

FastAPI → OpenAI API 동기 호출, 응답 완료까지 클라이언트 커넥션 점유 및 45초+ 대기

실패 시 전체 재실행

다단계 LLM 체인 중간 실패 시 앞 단계 결과 폐기, 비용과 시간 모두 낭비

📐 TO-BE Architecture
Temporal Workflow

LLM 호출을 Activity 단위로 분리, 중간 실패 시 해당 단계만 자동 재시도 — 앞 단계 결과 보존

LangChain 체인

프롬프트 → LLM(OpenAI) → 후처리 체인 구성, LCEL 파이프라인으로 선언적 조합

즉시 응답 + Redis Pub/Sub

요청 즉시 Channel Key 반환, 처리 상태를 Pub/Sub으로 실시간 스트리밍 — 클라이언트 블로킹 제거

🔧 기술 선택 근거
왜 Temporal?

NetCDF 프로젝트에서 검증된 패턴 재활용. 장시간 LLM 호출의 재시도·타임아웃을 선언적으로 관리 가능.

왜 Redis Pub/Sub?

SSE 대비 다중 클라이언트 구독이 자유롭고, Channel Key만 반환하여 커넥션 점유 제거.

크론으로 .sh 실행 → SQL 쿼리로 변경 데이터 비교 후 적재, 스크립트 유지보수 부담 및 장애 시 수동 대응 필요

🔧 기술 선택 근거
왜 Airbyte?

커넥터 기반 EL(Extract/Load) 자동화, 스케줄 Sync로 정합성 보장. Airflow 대비 설정이 간단.

왜 Elasticsearch?

웹툰 검색은 텍스트 검색·필터 위주 패턴. PostgreSQL LIKE 대비 역인덱스 기반 검색 성능이 압도적으로 빠름.
