Project 1 데이터 파이프라인 · 성능 최적화

NetCDF 데이터 처리 파이프라인 최적화

FastAPI SSE 엔드포인트 → Temporal Workflow 전환

문제 (AS-IS)

  • 대용량 NetCDF 응답을 클라이언트가 한 번에 수신 불가, SSE 커넥션 타임아웃 발생
  • 동시 요청 증가 시 단일 서비스에 병목 집중, 응답 지연 및 장애 전파
  • 데이터 가공 도중 에러 발생 시 전체 파이프라인 실패, 재시도 로직 부재

해결 (TO-BE)

  • Redis에 가공 완료 데이터를 적재, 클라이언트가 SSE로 순차 수신하도록 구조 변경
  • HPA 오토스케일링으로 Worker Pod 자동 확장, 병목 해소
  • Temporal Workflow 도입으로 단계별 자동 재시도 및 장애 복구 보장

성과

  • Redis 적재 + SSE 순차 수신으로 타임아웃 제거, 대용량 응답 안정적 전달
  • HPA 기반 Worker 자동 확장으로 동시 요청 수평 스케일링 가능한 구조 전환
  • Temporal 자동 재시도로 중간 실패 시 해당 단계부터 재개, 처리 유실 제거

Project 2 데이터 동기화 · 이벤트 기반

CDC 데이터 파이프라인 구축

Debezium Embedded Engine 기반 경량 CDC 구현

문제 (AS-IS)

  • @Scheduled 기반 1분 주기 DB 폴링으로 불필요한 부하
  • N개 테이블 × 24h × 60min = 하루 수만 건 SELECT 쿼리
  • 변경이 없어도 매분 전체 조회, 실제 감지까지 최대 1분 지연

해결 (TO-BE)

  • Spring Boot에 Debezium Embedded Engine 내장
  • PostgreSQL WAL을 앱 내에서 직접 캡처 · 처리
  • 별도 Kafka 인프라 없이 경량 CDC 구현

성과

  • 폴링 쿼리 100% 제거, DB 부하 대폭 감소
  • 변경 감지 지연 1분 → 실시간 (ms 단위)
  • Kafka 없이 단일 앱으로 운영, 인프라 비용 절감

Project 3 LLM 오케스트레이션 · 실시간 스트리밍 · 진행 중

LLM 기반 워크플로우 오케스트레이션 시스템

FastAPI + Temporal + LangChain + Redis 기반 LLM 요청 파이프라인

문제 (AS-IS)

  • LLM API 호출은 응답 시간이 길고 불안정 — 타임아웃, 토큰 제한, Rate Limit 등
  • 단순 동기 호출 시 클라이언트가 응답 완료까지 대기, UX 저하
  • LLM 체인이 복잡해질수록 중간 실패 시 전체 재실행 필요

해결 (TO-BE)

  • Temporal Workflow로 LLM 호출을 Activity 단위로 분리, 자동 재시도 보장
  • LangChain으로 프롬프트 → LLM(OpenAI) → 후처리 체인 구성
  • Redis Pub/Sub로 처리 상태를 실시간 스트리밍, 클라이언트 즉시 피드백

기대 효과

  • Temporal 재시도로 LLM API 장애 시 자동 복구, 안정성 확보
  • Redis 실시간 알림으로 "처리 중 → 완료" 상태를 즉시 전달
  • 체인 단계별 분리로 LLM 교체·확장 용이한 구조
